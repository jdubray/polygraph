# Subscription billing example

A synthetic worked example of the bare-next() method: a subscription billing
state machine (`source.cjs`) with states
`trialing`/`active`/`past_due`/`canceled`, actions `TRIAL_END`/`CHARGE`/
`RETRY`/`CANCEL`, and a dunning retry counter. It exercises the whole
Polygraph path -- controls, replay, and invariant model-checking -- **without
needing an API key**, the same way `examples/turnstile` does.

Related, deliberately different in purpose:

- `eval/machines/m01-subscription-renewal` -- a bench fixture (used by
  `npm run eval:check`) with a similarly-shaped seeded bug (a transient 5xx
  mishandled on one path but not its sibling). That one is scored, not walked
  through.
- `examples/case-study-subscription.md` -- a narrative writeup of a real,
  full-loop run against closed-source production code (source not included).

This example is a runnable, self-contained demo -- point people at it who want
to *run* the method themselves in under a minute, not just read about it.

## The seeded bug

`source.cjs` handles an ambiguous gateway error (`result: 'error'`) correctly
in the dunning path (`RETRY`): it's transient, so the row is left unchanged.
The initial charge path (`CHARGE`) is structured the same way but is
**missing that guard** -- any non-`ok` result, including a transient error,
falls through to `past_due`. A reader pattern-matching the two parallel charge
paths would expect both to no-op on `error`; the code does not.

This is a **sibling-path asymmetry**: a special case handled in one path,
silently missing in a parallel-looking sibling path. It's a common, realistic
defect shape -- and a good illustration of why replay alone isn't enough (see
below).

## Files

- `source.cjs` -- the "production code" under audit. `.cjs` (not `.js`)
  because `gen-traces.mjs` `require()`s it directly to derive traces, and this
  package is `"type": "module"`.
- `contract.json` -- observable state (`status`, `retries`), actions, and the
  special rule the seeded bug violates.
- `invariants.mjs` -- the invariant `no-past-due-on-transient-error`, encoding
  the intent the bug violates.
- `traces/*.ndjson` -- 22 windows across 6 scenarios, generated by driving
  `source.cjs` itself (`gen-traces.mjs`) so they're ground truth, not
  hand-typed. Every scenario is driven to the declared terminal state
  (`canceled`), per `validate_corpus.mjs`.
- `specs/reference.js` -- hand-written reference derived from `source.cjs`
  (the positive control). It **faithfully reproduces the seeded bug**.
- `specs-mutant/mutant.js` -- reference with the dunning cancellation
  threshold broken (3 retries -> 2), the negative control.

## Commands

From the plugin root:

```bash
# Regenerate the trace corpus from source.cjs (ground truth).
node examples/subscription/gen-traces.mjs

# 0. Validate the corpus (chaining, terminals, coverage table).
node scripts/validate_corpus.mjs \
  examples/subscription/contract.json examples/subscription/traces

# 1. Positive control -- reference must be fully consistent (22/22) on replay.
node scripts/verify.mjs \
  --contract examples/subscription/contract.json \
  --traces examples/subscription/traces \
  --specs examples/subscription/specs \
  --invariants examples/subscription/invariants.mjs \
  --out /tmp/sub-ref

# 2. Negative control -- mutant produces exactly 1 code-finding (the broken
#    cancellation threshold in s5_dunning_to_cancel).
node scripts/verify.mjs \
  --contract examples/subscription/contract.json \
  --traces examples/subscription/traces \
  --specs examples/subscription/specs-mutant \
  --out /tmp/sub-mut

# 3. (Optional) Real generation -- needs ANTHROPIC_API_KEY.
node scripts/verify.mjs \
  --contract examples/subscription/contract.json \
  --source examples/subscription/source.cjs \
  --traces examples/subscription/traces \
  --model sonnet-5 --n 5 --out /tmp/sub-gen
```

## Why this demo makes the pitch's point

Run 1 (the reference) comes back **22/22 consistent** on replay -- the derived
spec reproduces the code perfectly. That is *not* a clean bill of health: a
faithful spec reproduces bugs too, so replay alone can't see this one. It's
Part 2 -- iterating the spec against `invariants.mjs` -- that reaches the bad
state and reports it, with a counterexample:

```
no-past-due-on-transient-error (all-specs)
  init -> TRIAL_END({}) -> CHARGE({"result":"error"})
```

That's the loop: generate the spec from the code, propose/accept invariants,
then iterate the spec to find what replay alone would miss.
