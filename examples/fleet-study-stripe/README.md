# Fleet study · Tier 2 — the Stripe subscription fleet

Tier 2 of [`docs/fleet-study-plan.md`](../../docs/fleet-study-plan.md): a
fleet that accumulates **heterogeneously and under time** on a real public
API, so the paper can report what `polyvers` costs and finds against
something other than a fixture.

```bash
# no key: exercises the whole pipeline, produces a SIMULATED corpus
node examples/fleet-study-stripe/capture/capture.mjs --offline --fleet 400 --seed 7

# real Stripe test mode + Test Clocks (needs STRIPE_SECRET_KEY=sk_test_...)
node examples/fleet-study-stripe/capture/capture.mjs --live --fleet 400 --seed 7
```

Output lands in `out/`: `fleet.json` (the snapshot corpus for `polyvers
check --snapshots`), `traces/*.ndjson` (Polygraph replay windows), and
`manifest.json` (provenance).

## The projection bound — read this before any number

The contract declares **four** observable keys, and they are the entire
subject of every guarantee anything here makes:

| key | meaning |
|---|---|
| `subState` | Stripe's subscription status, camelCased |
| `dunningAttempts` | retries burned against the open invoice, 0..3 |
| `hasPaymentMethod` | a chargeable payment method is on file |
| `cents` | the recurring amount last settled or agreed |

**Everything else about a Stripe subscription is outside the projection and
is therefore not checked, not migrated, and not audited**: proration, tax,
invoice line items, payment-method fingerprints and brands, coupons and
discounts, currency, billing thresholds, schedules and phases, customer
metadata. A defect living purely in an unprojected field is invisible here
*by construction*, not by oversight.

This is the bound the paper must state explicitly. It is not a limitation of
the study design — it is what "verified" means in this toolset, and the
honest form of the claim is always *"verified with respect to this
projection and these invariants."*

## Two modes, and why the distinction is load-bearing

| | `--offline` | `--live` |
|---|---|---|
| needs a key | no | yes (test mode only; refuses a non-`sk_test_` key) |
| fleet is | generated by the verified machine | real Stripe API state |
| aging | simulated | Test Clocks (deterministic, real billing engine) |
| **admissible as Tier 2 evidence** | **no** | **yes** |

The offline corpus is **circular as evidence**: the machine generated it, so
of course the machine agrees with it. It exists to exercise the pipeline
end-to-end without a key — mapping, capture, export, validation — so the
live run is a contained change rather than a first attempt.
`manifest.admissibleAsTier2` records which one you have, and the Tier 3
report must refuse an offline corpus.

## What the capture is designed to catch

A fleet export is a snapshot at **one moment**, so subscriptions are caught
mid-flight. Two design choices follow, and both were forced by looking at
the first capture's output rather than by theory:

- **Trajectories are cut at a seeded point.** Running every subscription to
  completion produced a corpus of *settled* records — 3 distinct states from
  300 subscriptions. Cutting mid-flight yields 11 distinct states spanning
  every status, including the awkward ones (`incomplete`, mid-dunning
  `pastDue` at depths 1–2, `unpaid`, trial-lapsed-without-card) that are
  exactly where a version change hurts.
- **Stale deliveries are included.** At-least-once webhook delivery is not
  hypothetical: Stripe redelivers, and a final invoice event routinely lands
  *after* a subscription has ended. The first capture exercised **zero** of
  the contract's four named rejection rules; it now exercises all four. A
  corpus without stale deliveries under-tests precisely what the
  cross-version stimuli gate exists to check.

Current offline capture (`--fleet 400 --seed 7`): 400 subscriptions, ~1,360
windows, 11 distinct fleet states, all four rejection rules covered.

## Validating a capture

```bash
M=examples/fleet-study-stripe/machines/subscription-v1
O=examples/fleet-study-stripe/out

# the corpus is well-formed AND no captured state can be driven to a violation
node scripts/check.mjs --spec $M/next.cjs --contract $M/contract.json \
  --invariants $M/invariants.mjs --initial-states $O/fleet.json

# trace corpus shape + special-rule coverage
node scripts/validate_corpus.mjs $M/contract.json $O/traces
```

`validate_corpus` will report chaining/terminal **warnings** on a fleet
corpus and exit nonzero. That is expected and not a defect: its terminal
check assumes a trace is a complete scenario, whereas fleet traces are
truncated mid-flight by construction. Read the special-rule coverage and the
chain integrity; ignore the terminal warnings for this corpus.

## A bug this capture found

`validate_corpus`'s special-rule tally compared `whenState` to a bare state
value with `===`, so it never matched the *expression* form that
polygen-authored contracts use (`"orderState == 'charging'"`). Every such
contract silently scored **0 windows for every rule** and warned about it —
including the repo's own demo corpus, which contains traces named
`cancel_blocked_while_charging_*`. Fixed in `scripts/validate_corpus.mjs`:
both conventions (bare literal and expression) are now matched, and rules
whose `whenState` is prose are reported as *not mechanically measurable*
rather than as zero coverage.
