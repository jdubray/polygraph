{intent_domains}

  - `acceptors` — an OBJECT keyed by the same action names (the framework
    binds proposals to acceptors; do NOT write flag guards or a dispatch
    switch). Each acceptor is
    `(model) => (proposal, { reject, next, unchanged }) => { ... }` — a
    NEXT-STATE relation (sam-pattern 2.1 prime semantics): `model` is the
    FROZEN pre-state (read-only; writing `model.x` throws), and every write
    goes to the `next` draft.
  - COMMIT RULE (next-state form): for a FLAT state key, assign
    `next.someKey = value` — the right-hand side reads the PRE-state
    (`next.count = model.count + 1`). For a state key holding a NESTED
    map/object, assign the key WHOLE:
    `next.<key> = { ...model.<key>, [k]: updated };` — do NOT mutate
    `model.<key>[k].field` or `next.<key>[k].field` in place.
  - FRAME RULE: on every ACCEPTED path, every declared modelShape key must be
    either assigned via `next.<key> = …` or explicitly named unchanged:
    `unchanged('keyA', 'keyB')` — otherwise the step throws `SamFrameError`.
    A rejected step (`return reject(reason)`) needs NO framing. Never assign
    the same key twice in one step.
  - REJECT MEANS DECLINED, FULL STOP: `reject(reason)` declares "this
    proposal does NOT apply — the step is a no-op (post == pre)". It is the
    TERMINAL statement of a branch that changes nothing, and it MUST NOT be
    preceded by any `next.*` write in that branch — a reject DISCARDS the
    entire `next` draft, silently throwing away work. There is NO
    "accept with a reason" primitive: never call `reject(reason)` to tag,
    annotate, or explain a SUCCESSFUL transition (a branch that assigns
    `next.*` simply returns after its writes; the special-rule names below
    are reasons for REJECTIONS only, not labels for accepted branches).

## In-context example: Rocket Launcher in the v2 strict profile

```javascript
'use strict';

const { createInstance } = require('@cognitive-fab/sam-pattern');

const instance = createInstance({ strict: true, hasAsyncActions: false });

// status: 'ready' | 'counting' | 'aborted' | 'launched'
const INITIAL_STATE = { counter: 10, status: 'ready' };

const control = instance({
  initialState: { ...INITIAL_STATE },
  component: {
    modelShape: {
      counter: { type: 'number' },
      status: { type: 'string' },
    },
    actions: {
      START: { action: (data = {}) => ({ ...data }), schema: {}, domain: [{}] },
      DECREMENT: { action: (data = {}) => ({ ...data }), schema: {}, domain: [{}] },
      ABORT: { action: (data = {}) => ({ ...data }), schema: {}, domain: [{}] },
    },
    acceptors: {
      START: (model) => (proposal, { reject, next, unchanged }) => {
        if (model.status !== 'ready') return reject('start-only-when-ready');
        next.status = 'counting';
        unchanged('counter');
      },
      DECREMENT: (model) => (proposal, { reject, next, unchanged }) => {
        if (model.status !== 'counting') return reject('not-counting');
        const counter = model.counter - 1;   // RHS reads the PRE-state
        next.counter = counter;
        if (counter === 0) next.status = 'launched';
        else unchanged('status');
      },
      ABORT: (model) => (proposal, { reject, next, unchanged }) => {
        if (model.status !== 'counting') return reject('nothing-to-abort');
        next.status = 'aborted';
        unchanged('counter');
      },
    },
    reactors: [],
  },
});

const { intents } = control;

const getState = () => instance({}).getState();
const setState = (snapshot) => { instance({}).setState(snapshot); };

// NOTE: do not use instance({}).state() anywhere — getState() is the safe
// snapshot accessor (a model data key like 'state' would return data, not the
// method; see sam-lib #29). init() is EXACTLY this one call:
const init = () => { setState(INITIAL_STATE); };

const actions = {
  START: (data = {}) => intents.START(data),
  DECREMENT: (data = {}) => intents.DECREMENT(data),
  ABORT: (data = {}) => intents.ABORT(data),
};

module.exports = { instance, init, actions, getState, setState };
```

(The rocket's state is flat, so its acceptors assign model keys directly. When
a state key holds a nested map/object, commit each change with the top-level
write `model.<key> = { ...model.<key>, [k]: updated }` instead.)
